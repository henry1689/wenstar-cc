# 模块审计流程 · Module Audit Protocol

## 触发条件

当对任一模块(M1-M9/app/webui）进行深度审查时，严格遵循以下五步流程。**不允许跳步。**

---

## 第一步：阅读设计意图

在阅读任何代码之前，先找到该模块的**原始设计文档**：

```
src/m*/<module>/          → 文件注释 + 导出
ARCH.md                   → 架构设计文档
src/m*/types/             → 接口定义
```

回答以下问题：
1. 这个模块的**核心定位**是什么？（一句话）
2. 它的**输入和输出**是什么？
3. 它的**边界约束**是什么？（不该做什么）
4. 它和其他模块的**契约**是什么？（接口/类型依赖）

---

## 第二步：审查实现

逐文件审查，关注：

### 2.1 边界合规
- [ ] 是否做了不该做的事？（如 M1 不应有钙化计算/LLM调用）
- [ ] 是否依赖了不该依赖的模块？（如 M1 不应 import M3/M5）
- [ ] 导出符号是否被外部合理使用？（grep 跨模块引用）

### 2.2 实现质量
- [ ] 硬编码配置是否有外部化？（路径/词表/阈值）
- [ ] 是否有重复逻辑？（同一套规则在两个地方实现）
- [ ] 类型安全：有无 `as any` 逃逸？
- [ ] 错误处理：catch 后是否静默吞错误？

### 2.3 数据流追踪
- [ ] 数据是否流入预期消费者？
- [ ] 是否有数据"泄漏"到不该去的模块？
- [ ] 是否有死代码（导出但无人引用）？

---

## 第三步：决策

对发现的问题分级：

| 级别 | 定义 | 处理方式 |
|------|------|---------|
| **🔴 P0** | 偏离原设计方向 / 数据损坏风险 / 类型逃逸 | **必须立即修复** |
| **🟡 P1** | 违反了边界约束 / 词表漂移 / 设计不一致 | **在同一轮次修复** |
| **🔵 P2** | 硬编码/重复逻辑/无缓存 | **记录到 issue/技术债清单，可后续修复** |
| **⚪ P3** | 注释过时/命名不一致/可读性 | **记录下来即可** |

关键原则：
- **🔴 P0 和 🟡 P1 必须在本轮修复**，不修完不进入下一模块
- 🔵 P2 如果修复成本低（<5分钟），也当场修掉
- 修复后必须 `npx tsc --noEmit` 编译验证

---

## 第四步：修复实施

```
1. 写修复代码
2. npx tsc --noEmit --pretty     ← 编译验证
3. 检查如果有运行中服务器：相关测试
4. 更新稳定性检查技能（如需要）
```

---

## 第五步：模块结论

在返回给用户前，给出：

```
## M<N> — <模块名> · 审计结论

### 原始定位
一句话定位

### 发现的问题
| # | 级别 | 问题 | 修复 |
|---|------|------|------|
| 1 | P1 | ... | ✅ 已修复 |

### 方向判定
🟢 方向正确 / 🟡 轻微偏离 / 🔴 方向错误

### 下一模块
→ M<N+1>
```

---

## 第六步：验证收尾

- [ ] 编译零错误
- [ ] P0/P1 全部修复
- [ ] 结论写入模块日志
- [ ] 进入下一模块
