# 标准变更报告格式

## 状态：🟢 已自动常驻（无需调用）

此规则已写入 `CLAUDE.md` 永久行为规则第 2 条，每次改完代码自动输出。**不再需要手动调用。**

## 使用时机

## 报告模板

```markdown
## 变更报告

### 改了什么
- 列出每个修改的文件和具体改动（1-2句/条）
- 包括新增/修改/删除的文件

### 为什么要改
- 说明原始设计的意图（为什么之前那样写）
- 说明当前的问题或Bug根因（为什么需要改）
- 说明这个根因是怎么发现的

### 改了的后果
- ✅ 正面影响：修复了什么、提升了什么
- ⚠️ 副作用：有无数据损失、兼容问题、功能删减
- 🔄 是否需要迁移/配置变更
- 📦 影响范围：哪些模块/功能会受影响

### 是否需要回复
- 是/否
- 如果"是"，说明需要用户确认什么（如：是否同意此方案、是否恢复某功能等）
```

## 示例

详见一次典型输出：

### 改了什么
**文件**：`src/webui/server.ts`

**两处改动**：
1. 初始化 `conversationDB`（`initPipeline()` 内）
2. `loadConversationHistory()` 读取路径从 `storage.getSQLite()` 改为 `conversationDB`

### 为什么要改
**根因：写入和读取走的不是同一个数据库。**

`conversationDB` 变量声明为 `ConversationDB | undefined`，**从未被初始化**（没有 `new ConversationDB()`），导致：
- 写入：`ctx.conversationDB?.insertConversation()` — `?.` 静默跳过，什么都没写
- 读取：`storage.getSQLite().getRecentConversations()` — 目标数据库无 `conversations` 表
- 仅内存数组 `conversationHistory` 生效，刷新即消失

### 改了的后果
- ✅ 刷新后对话保留
- ⚠️ 之前未正确落盘的旧数据不可恢复
- ✅ 其他功能无影响

### 是否需要回复
否
```

## 原则
1. 用户已明确要求的处理方式，如无新问题不再重复询问
2. 如果修改有潜在风险（数据丢失、功能删减、API变更），必须在"后果"中明确说明
3. 如果修改涉及用户偏好或需要决策，必须在"是否需要回复"中说明理由
4. 用户同意后才能在代码中落地
