# ADR-003: chat.ts 分层拆分策略

- **状态**: 已接受 — 已实施
- **决策**: 将 2734 行的 chat.ts 按功能阶段拆分为独立 stage 文件
- **日期**: 2026-06-30

## 背景

`src/webui/chat.ts` 是系统的核心聊天管线，最初为 2734 行单文件。审计发现：
- 30+ 个下划线前缀变量导致命名污染
- try 嵌套深达 4 层
- 模块级全局变量在函数内多处读写，改一处要理解全部代码

## 决策

### 拆分模式

采用 wenstar-cx 的 stage 模式：每个独立功能抽取为单独文件，主文件只做编排。

```
chat.ts → 编排器（~2288行，保留核心管线）
chat/dialog-group-stage.ts  → 闭组写入（139行）
chat/persistence-stage.ts   → 对话持久化（86行）
chat/retrieval-stage.ts     → 记忆检索+黑钻（350行）
chat/guard-builder.ts       → 守卫构建（已有）
chat/post-process.ts        → 后处理（已有）
chat/retrieval.ts           → 外部检索（已有）
chat/index.ts               → 统一导出（已有）
```

### 函数签名约定

每个 stage 函数：
- 使用单个对象参数（wenstar-cx 风格）
- 返回结构化数据
- 不依赖模块级全局变量（通过参数注入）

## 理由

1. **渐进安全**: 采用"不删旧代码，只加新代码"策略，旧路径继续运行。
2. **分步验证**: 每次提取后用 diff 对比原版函数体，逐行验证一致性。
3. **风险控制**: 保留在 chat.ts 的核心管线（2288行）不再拆分——变量相互引用，拆了反而更难维护。

## 后果

- 拆出 3 个独立文件，共 -446 行
- 函数体 100% 一致验证通过
- 零新增测试失败
- 剩余 ~2288 行核心管线保留在原文件
