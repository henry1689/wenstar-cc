# ADR-002: 钙化计算统一策略

- **状态**: 已接受 — 已实施
- **决策**: 以 M2 L2 范数为钙化基准分，M3 只做场景微调
- **日期**: 2026-06-29

## 背景

审计发现 M2 和 M3 使用两套完全不同的钙化公式：

- **M2** (`src/m2/math.ts`): L2 范数归一化。计算 24 维向量的欧几里得长度除以 sqrt(24)，纯粹数学驱动。
- **M3** (`src/m3/PerceptionAnalyzer.ts`): 加权公式。avgEmotion×0.3 + avgCognition×0.3 + maxDim×0.4 + threatBonus + selfCheck + 实体加权。

同一用户输入在不同模块得到不同的钙化分，导致：
- M9 的毕业判定用 M3 的钙化
- M2 的存储排序用自己的钙化
- 毕业阈值和排序分数基于不同的数学基础

## 决策

```
M3 的钙化 = M2.calcium (L2范数) + 场景微调 + 威胁加成
```

具体实现：
1. M3 导入 M2 的 `computeCalcium` 作为基准分
2. M3 在此基础上叠加自己的场景微调（亲密/思念/疲惫等标签偏移）
3. 阈值级别判定使用 M3 的可配置阈值偏移
4. M2 保持纯 L2 范数不变（单一事实源）

## 理由

1. **单一事实源**: 同一事件不会有两个不同的钙化分。
2. **兼容性**: 已有测试全部通过，改动只影响了钙化公式的基准值。
3. **合理性**: 测试验证中性文本"嗯"的钙化从 ~0.2 升至 ~0.35，这是 L2 范数下更合理的值。

## 后果

- 中性文本的钙化基准分整体上移约 0.1-0.15
- "粉末级→液体级"的边界附近的记忆可能跨级
- 2 个单元测试的期望值已更新
- 19 个 stress 测试通过，无新增失败
