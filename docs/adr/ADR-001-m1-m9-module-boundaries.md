# ADR-001: M1-M9 模块边界定义

- **状态**: 已接受 — 已实施
- **决策**: M1-M9 严格分层，禁止跨层调用
- **日期**: 2026-06-02（最初审定），2026-06-29（本轮审计确认）

## 背景

系统采用类脑认知架构，按功能分为 9 个模块 + 应用层。如果没有明确的边界约束，模块间容易产生循环依赖和职责混淆——M1 写入了 M3 的逻辑，M5 反向引用了 M8 的数据。

参考：M2 设计阶段因 M1 越界做 M3 的感知计算而产生的架构纠偏，导致 `PerceptionAnalyzer.ts` 从 M1 迁移至 M3。

## 决策

### 模块依赖方向（单向）

```
M1 → M2 → M3 → M4 → M5 → M6/M7/M8/M9
```

- **M1 (DNA编码)**: 零外部依赖。只接受原始输入，输出 DNA 对象。不得调用 M3 的感知计算，不得调用 M5 的表达生成。
- **M2 (融合存储)**: 依赖 M1 的 DNA 类型。只做持久化和检索，不做决策。不得调用 M3 的钙化公式。
- **M3 (感知决策)**: 依赖 M1 的 DNA 类型 + M2 的检索常量。只做感知分析和决策，不做表达。不得调用 M5 的 LLM。
- **M4 (知识融合)**: 依赖 M2 的检索接口 + M3 的决策类型。只做知识融合和人物画像，不直接访问存储。
- **M5 (表达生成)**: 依赖 M3/M4 的输出类型 + M8/M9 的接口（已存在反向依赖 P0，待修复）。只做回复生成，不做存储。
- **M6-M9**: 依赖 M2 的存储接口，不依赖 M5 的输出。独立演化模块。

### 已发现的违规

| 违规 | 级别 | 说明 | 修复 |
|------|------|------|------|
| M5 → M8 反向依赖 | P0 | M5ClueAssistant 调用 M8 derivePhysiologicalSnapshot | 抽象为接口注入 |
| M5 → M9 反向依赖 | P0 | DeepSeekLLMProvider 动态 import M9 WorkingMemory | 移除 |
| M3 → M1 反向依赖 | P1 | PerceptionAnalyzer import M1 LexiconLoader.loadSet | 将词表移至共享目录 |
| M9 → M3 反向依赖 | P1 | WorkingMemory 调用 PerceptionAnalyzer.recalculateCalcium | 统一钙化后移除 |

## 理由

1. **可测试性**: 单向依赖意味着每个模块可独立 mock 测试。
2. **可维护性**: 改 M3 的感知算法不影响 M1 的编码逻辑。
3. **可替换性**: 如果将来要替换 M5 的 LLM 提供商，不需要改 M3 的感知逻辑。

## 后果

- 拆分了 chat.ts 中的 3 个独立模块（闭组写入/持久化/检索），验证通过
- 钙化算法已统一（M2 L2 范数为基准，M3 只做场景微调）
- 剩余 P0/P1 违规待修复
