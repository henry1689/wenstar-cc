## 二、DNA 编码体系（v2.0 物料化改造）

### 2.1 根码格式

```
DNA-YYYYMMDD-HHmm-NNNN-HX
│    │        │    │    │
│    │        │    │    └─ HY 瑶印码（1 位 hex，源自用户 ID + 时间戳哈希）
│    │        │    └─────── 4 位流水号（每日 0000-9999 循环）
│    │        └──────────── 分钟（24 小时制）
│    └───────────────────── 日期
└────────────────────────── DNA 静态前缀
```

**总长度**: `DNA-`(4) + `YYYYMMDD`(8) + `-`(1) + `HHmm`(4) + `-`(1) + `NNNN`(4) + `-`(1) + `HX`(1) = **24 字符**

### 2.2 瑶印码生成算法

```typescript
function generateHYStamp(userId: string, timestamp: Date): string {
  const hash = simpleHash(`${userId}:${timestamp.toISOString().slice(0, 10)}`);
  return (hash % 16).toString(16).toUpperCase(); // 0-F
}
```

- `simpleHash`: 轻量非加密哈希（FNV-1a 或 DJB2），不引入 crypto 依赖
- 输入：`用户ID + 当天日期`
- 输出：1 位 hex（0-9, A-F）
- 同一用户同一天产生相同瑶印码，用于数据泄露溯源

### 2.3 环节特征码体系

每个模块在 DNA 中附加 `sub_id`，格式：

```
[DNA根码].[模块代码].[流水号]
```

| 模块 | 代码 | 示例 |
|------|------|------|
| M1 感知 | M1 | `DNA-20260623-1430-0001-A.M1.001` |
| M3 24D 感知 | M3 | `DNA-20260623-1430-0001-A.M3.001` |
| 记忆存储 | MEM | `DNA-20260623-1430-0001-A.MEM.001` |
| 黑钻晋升 | BD | `DNA-20260623-1430-0001-A.BD.001` |
| 人物图谱 | GRAPH | `DNA-20260623-1430-0001-A.GRAPH.001` |

## 三、三段式存储晋升体系

### 3.1 存储层定义

| 层级 | 数据库表 | 物理文件 | 保留策略 | 可压缩 |
|------|----------|----------|----------|--------|
| 砂金库 (Alluvial) | `conversations` | `hermes.db` | 永久保留原文 | 30 天后可压缩 |
| 金库 (Gold) | `memories` | `hermes.db` | 随钙化分衰减 | 衰减后可清除 |
| 黑钻库 (Black Diamond) | `black_diamond` | `hermes.db` | 永久保留 | 不可压缩 |

### 3.2 钙化分（Calcium Score）规则

**定义**: 一段对话或记忆的情感价值量化指标，范围 [0, 10]

**初始计算** (M3 产出):

```
calcium = round(
    emotional_intensity * 0.30 +
    intimacy_level * 0.25 +
    social_relevance * 0.20 +
    factual_density * 0.15 +
    unusualness * 0.10
, 1)
```

其中 `emotional_intensity` = max(所有 24 维情感值的绝对值) / 10

**动态增长**:

```
每次被召回： calcium = min(10, calcium + 0.2)
```

**衰减规则** (每 24 小时执行):

```
中性情感 (valence ≈ 5):
  work_related → calcium = max(0, calcium - 0.05)
  personal     → calcium = max(0, calcium - 0.10)

强烈情感 (valence 偏离 5 > 2):
  calcium = max(0, calcium - 0.02)  // 衰减极慢

黑钻记忆: 不衰减
```

### 3.3 晋升规则

**砂金 → 金库** (每 30 分钟评估):

```
条件: calcium >= 1
动作:
  1. 从 conversations 表读取原文
  2. 调用 M3 重新生成 24D 感知
  3. 写入 memories 表
  4. 标记 conversations.is_promoted = 1
```

**金库 → 黑钻** (每 2 小时评估):

```
条件: calcium >= 4.5  OR  recall_count >= 5
动作:
  1. 从 memories 表读取语义块
  2. 重新计算情感向量
  3. 写入 black_diamond 表（含 FTS5 全文索引）
  4. 标记 memories.is_promoted = 1
```

### 3.4 黑钻库 FTS5 全文索引

```sql
-- 创建 FTS5 虚拟表
CREATE VIRTUAL TABLE black_diamond_fts USING fts5(
    content, summary, tags,
    content='black_diamond',
    content_rowid='rowid',
    tokenize='unicode61'
);

-- 触发器维持同步
CREATE TRIGGER black_diamond_ai AFTER INSERT ON black_diamond BEGIN
    INSERT INTO black_diamond_fts(rowid, content, summary, tags)
    VALUES (new.rowid, new.content, new.summary, COALESCE(new.tags, ''));
END;

-- 检索查询
SELECT bd.* FROM black_diamond bd
JOIN black_diamond_fts fts ON bd.rowid = fts.rowid
WHERE black_diamond_fts MATCH ?;
```

## 四、VAD 谱曲引擎健康缓存

### 4.1 缓存机制

```typescript
// 模块级变量（非实例级，全局生效）
let _vadAvailable: boolean = true;

async function checkVadHealth(): Promise<void> {
  if (!_vadAvailable) return; // 永久标记不可用
  try {
    await fetch('http://localhost:8100/health', { signal: AbortSignal.timeout(3000) });
  } catch {
    _vadAvailable = false;
    logger.warn('[VAD] Composer offline, permanently disabled for this session');
  }
}
```

### 4.2 重置端点

```typescript
// GET /api/admin/reset-vad
async function handleResetVad(req: http.IncomingMessage, res: http.ServerResponse): Promise<void> {
  _vadAvailable = true; // 或使用某种动态恢复检测
  await checkVadHealth(); // 立即检测一次
  res.end(JSON.stringify({ status: 'ok', vadAvailable: _vadAvailable }));
}
```

### 4.3 预期行为

| 场景 | 行为 |
|------|------|
| 8100 端口离线 | 首次消息提示超时，设置 `_vadAvailable = false` |
| 后续消息 | 无 VAD 日志输出，跳过所有 VAD 调用 |
| 每次响应节省 | ~2 秒（超时等待时间） |
| 调用重置端点 | 重新检测一次，如果在线则恢复 |

| 无人名对话（少见） | 2 次 API | 2 次 API | 不变 |

## 六、RoleGuard 输出校验与激活

### 6.1 校验流程

```
M5 生成原始输出
    │
    ├──→ RoleGuard.validate(output, currentRole)
    │         │
    │         ├── [校验通过] → 输出到用户
    │         │
    │         └── [校验失败]
    │               ├── [重试 < 2 次] → 回传 M5 重新生成
    │               │                    （带约束 prompt）
    │               │
    │               └── [重试 ≥ 2 次]
    │                     └── 降级为「秘书」角色
    │                         重新生成输出
    │                         并记录 WARN 日志
    │
    └──→ 输出到用户
```

### 6.2 角色校验规则

| 角色 | 拦截规则 | 强制规则 |
|------|----------|----------|
| 秘书 (secretary) | 亲密词黑名单（喘/腿/腰/蹭/抱/吻/摸/操/干/射等） | 无 |
| 回忆者 (recaller) | 推测词（应该/可能/大概/也许） | 必须包含「不记得」类表达 |
| 亲密模式 (lover) | 无拦截（允许所有表达） | 无 |
| 导师 (mentor) | 无拦截 | 必须包含建设性意见 |
| 同事 (colleague) | 工作无关的亲密表达 | 无 |

### 6.3 实现要点

```typescript
class RoleGuard {
  private retryCount: number = 0;
  private readonly MAX_RETRIES = 2;

  async validateAndFix(
    output: string,
    currentRole: string,
    regenerate: (constraint: string) => Promise<string>
  ): Promise<{ output: string; finalRole: string }> {
    const violations = this.checkViolations(output, currentRole);
    if (violations.length === 0) return { output, finalRole: currentRole };

    while (this.retryCount < this.MAX_RETRIES) {
      this.retryCount++;
      logger.warn(`[RoleGuard] Violation detected (${violations.join(', ')}), retry ${this.retryCount}/${this.MAX_RETRIES}`);

      output = await regenerate(`约束: ${violations.map(v => `禁止出现${v}`).join('; ')}`);
      const retryViolations = this.checkViolations(output, currentRole);
      if (retryViolations.length === 0) return { output, finalRole: currentRole };
    }

    // 降级为秘书
    const fallbackRole = 'secretary';
    logger.warn(`[RoleGuard] Max retries exceeded. Forcing fallback to ${fallbackRole}`);
    const fallbackOutput = await regenerate(`你现在的角色是${fallbackRole}。${VULGAR_BLOCK_PROMPT}`);
    return { output: fallbackOutput, finalRole: fallbackRole };
  }
}
```


### 7.2 禁止行为

- ❌ 不得引用 `src/` 主程序下的任何模块
- ❌ 不得修改主程序的数据库 schema
- ❌ 不得引入与核心能力无关的外部包
- ❌ 不得包含任何业务角色逻辑（玉瑶/秘书等）

### 7.3 允许行为

- ✅ 使用相同的数据规范（DNA 格式、钙化分公式、元标签字段）
- ✅ 通过 HTTP 3737 接口与主程序通信
- ✅ 读取主程序数据库（只读，通过独立连接）
- ✅ 在主程序数据库中写入词条数据（通过约定的接口表）
