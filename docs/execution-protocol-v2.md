## 第一章：协作流程（SOP）

### 1.1 标准开发闭环

每个模块的开发必须经过以下标准闭环，**不可跳过任何步骤**:

```
需求/任务明确
    │
    ▼
① 复述核心约束 ──── 开始编码前，向决策者复述该模块在规格书中的核心约束与Hook要点
    │
    ▼
② 输出设计文档 ──── 包括接口定义、数据结构、关键决策点
    │
    ▼
③ 设计评审/确认 ──── 决策者确认设计方向正确
    │
    ▼
④ 编码实现
    │
    ▼
⑤ 自检自修 ──── 编码 → 运行Hook → 分析失败 → 自动修复 → 重跑直至通过
    │
    ▼
⑥ Hook验证 ──── 所有 Validation Hook 必须 100% 通过
    │
    ▼
⑦ 更新规格书 ──── 将实现写入规格书对应章节（新增或修订）
    │
    ▼
⑧ 总体验收 ──── 决策者确认交付物符合规格书
```

### 1.2 强制卡点

| # | 卡点名称 | 触发条件 | 后果 | 例外条款 |
| :--- | :--- | :--- | :--- | :--- |
| **G-01** | 无复述不编码 | 开始编码前未复述核心约束 | 决策者可要求重新开始 | 无例外 |
| **G-02** | 无设计不编码 | 未输出设计文档即开始写代码 | 拒绝进入编码阶段 | 紧急修复允许先写代码，但须在24h内补设计文档 |
| **G-03** | 无Hook不提交 | 未编写对应模块的 Validation Hook | Hook 补齐前不予验收 | 无例外 |
| **G-04** | 无文档不验收 | 模块交付时未更新规格书对应章节 | 不签字通过 | 无例外 |
| **G-05** | 三修不过必报告 | 连续3次自修仍无法通过 Hook | 暂停，48h内报告决策者 | 报告前不可再次尝试 |
| **G-06** | 无决议不停留 | 遇到需决策者确认的【待决事项】卡住超过1轮 | 必须升级为提问，不得自行假设 | 规格书和协议中已明确定义的细节不在"待决"范围 |

### 1.3 各角色的职责

**决策者**（系统所有者）:
- 对架构变更具有最终决定权
- 审阅并批准设计文档 → 编码的推进许可
- 审阅并批准模块交付物
- 对自修三连败做出人工介入决策
- 签署架构决策记录（ADR）

**实施者**（AI Agent / 开发者）:
- 严格遵守本协议的所有条款
- 在开始编码前复述模块约束
- 输出设计文档供决策者审阅
- 执行自检自修闭环
- 编写并运行 Validation Hook
- 更新规格书对应章节
- 标记所有模糊点（使用【待决事项】标签）

## 第二章：架构决策记录（ADR）规范

### 2.1 为什么需要 ADR

项目中会有大量架构决策：L0为什么不用LLM？M1-M2边界为什么这样切？这些决策如果只存在对话记录里，三个月后就丢失了上下文。ADR 就是为了解决这个问题。

### 2.2 文件规范

- **存放路径**: `docs/adr/`
- **文件命名**: `ADR-{NNN}-{kebab-case-slug}.md`
- **模板**:

```markdown
# ADR-{NNN}: {决策标题}

- **状态**: {提议 | 已接受 | 已废弃 | 已超驰}
- **决策日期**: {YYYY-MM-DD}
- **关联模块**: {M1 / M2 / 全局}

## 背景
{为什么需要做这个决策？面临什么问题？}

## 选项
{列出考虑的方案，每个一行}

## 决策
{最终选择的方案}

## 理由
{为什么选这个而不是其他？核心论据}

## 后果
{选了之后有什么影响？正面和负面}

## 关联文档
{关联的规格书章节、ADR编号等}
```

## 第三章：AI Agent 行为准则

### 3.1 遇到模糊需求时的处理流程

```
发现模糊/未定义点
    │
    ▼
① 在代码/文档中用 【待决事项】 标签标记
    │
    ▼
② 在对话中明确提问："关于X，我需要明确：选项A（……）还是选项B（……）？"
    │
    ▼
③ 决策者给出方向后，如果涉及规格书修改 → 先更新规格书 → 再继续编码
```

**两个"不"**:
- ❌ **不自行假设**——当规格书没有明确定义时，不猜测意图
- ❌ **不只提问不标记**——提问的同时必须在文档中留下【待决事项】标签

### 3.2 代码注释规范

| 注释类型 | 格式 | 示例 |
| :--- | :--- | :--- |
| 章节引用 | `// Ref: SPEC.md §3.2` | SPEC = 规格书文件名 |
| 架构决策引用 | `// Ref: ADR-001` | ADR 编号 |
| 关键逻辑说明 | `// 此处为何这样实现` | 不超过3行 |
| 自修记录 | `// [SELF-REPAIR] Hook X failed because Y. Fixed by Z per SPEC.md §N.` | 失败记录，不超过5行 |
| 【待决事项】 | `// 【待决事项 3-3】: 是否需要在 M1 增加运行时 Schema 校验 | 统一格式 |

### 3.3 禁止行为清单

以下行为属于**违规**，无论代码质量多高或功能多完整，都不可验收：

1. **绕开自检自修闭环** —— 编码后不运行 Hook 就提交
2. **未确认假设设计意图** —— 遇到模糊点不提问，直接选一个方案实现
3. **先写了再说** —— 在不确定设计方向时就开始编码
4. **跳过 Hook 编写** —— 模块实现完成后才想起补测试
5. **降低确定性标准** —— 在编码阶段引入任何形式的随机性（除非规格书明确要求）
6. **跨模块耦合** —— 在 M1 中写入 M2 的职责（如持久化逻辑）
7. **未经ADR直接修改已确权设计** —— 确权后的设计变更必须走 ADR 流程

### 3.4 需要人工确认的场景

以下场景**必须**主动请求决策者介入，不可由 Agent 自行推进：

| 场景 | 触发条件 | 需提供的材料 |
| :--- | :--- | :--- |
| 架构变更 | 涉及核心范式或锚点的修改 | ADR草案 + 影响分析 |
| 三修不过 | 连续3次自修无法通过Hook | 错误日志 + 失败分析 + 规格书对应章节引用 |
| 新模块启动 | 进入下一个 M{N} 模块前 | 前模块验收签字 + 新模块的核心约束复述 |
| 技术选型 | 需要决策存储方案/外部工具选型 | 备选方案对比表 |
| 偏差确认 | 发现代码必须偏离规格书 | 偏差原因 + 替代方案 |

### 4.2 代码规范

| 规范项 | 要求 | 违反后果 |
| :--- | :--- | :--- |
| **类型定义** | 所有公共接口必须有显式类型定义，禁止使用 `any` | Hook 不通过 |
| **JSDoc** | 所有公共方法必须有 JSDoc（用途+参数+返回值） | PR 退回 |
| **章节引用** | 关键逻辑必须有 `// Ref: SPEC.md §X.X` | 代码审查退回 |
| **模块边界** | M1 代码不可引用 M2+ 的模块 | 结构违规，不可验收 |
| **硬编码禁止** | 配置类数据必须从外部文件读取（taxonomy / self_model / keyword rules） | 代码审查退回 |
| **单文件行数** | 不超过 500 行（超过须说明理由） | 建议重构 |

## 第五章：变更控制

### 5.1 变更控制原则

```
代码与规格书冲突时 → 以规格书为准 → 修改代码使其符合规格书
规格书需要变更时 → 发起 ADR → 决策者审阅 → 批准后更新规格书 → 再更新代码
```

### 5.2 变更类型与流程

| 变更类型 | 定义 | 流程 | 审批人 |
| :--- | :--- | :--- | :--- |
| **微调** | 不涉及接口/架构/设计的优化（如变量命名、注释修正） | 直接实施，在提交时注明 | 无需审批 |
| **常规变更** | 涉及接口参数调整、新增配置项、测试用例补充 | 输出ADR → 决策者确认 → 实施 | 决策者 |
| **架构变更** | 涉及核心范式、五大锚点、模块边界重定义 | 输出ADR + 影响分析 → 决策者评审 → 确认 | 决策者 |
| **紧急修复** | 线上故障修复，需要立即修复 | 先改代码，24h内补ADR和规格书更新 | 事后审阅 |

### 5.3 版本号规则

本项目的版本号（规格书 / 协议 / 模块）遵循：

```
v{major}.{minor}
```

- **major**: 架构变更（违反向后兼容）
- **minor**: 功能新增或修复（向后兼容）

当前版本:
- 规格书: **v1.0**-draft
- 协议: **v2.0**-draft
- 项目整体: **v0.1.0**（pre-alpha）
