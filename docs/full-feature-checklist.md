## 一、核心记忆体系

| # | 功能点 | 说明 | 测试方法 |
|:-:|--------|------|---------|
| 1 | 三库分层存储 | 砂金库(conversations)、金库(memories)、黑钻库(black_diamond)三级分层 | 查三张表均存在，数据量级符合比例 |
| 2 | 砂金库永久留存 | 原始对话只标记 `is_compacted`，不物理删除 | 压缩后原文仍在，仅标记状态变化 |

| 4 | 黑钻永久记忆 | 钙化 ≥ 4.5 自动晋升，永久锁定不衰减 | 高情绪对话触发后 `black_diamond` 表新增记录 |
| 5 | 三段式关联 | `dna_root_id` / `dialog_group_id` / `seq_pos` 一一对应 | 新对话闭组后三段可通过ID联查，关联率100% |
| 6 | 记忆自动流转 | 砂金→金库(钙化≥1)→黑钻(钙化≥4.5/召回≥5) | 完整走一轮对话，三级自动流转无需干预 |
| 7 | 记事记忆子系统 | 复用金库表(`memory_type='note'`)，4类场景 | 查 `memories` 表可筛选出记事型记录 |
| 8 | 物品位置记忆 | 记住物品摆放位置，支持覆盖更新 | 说"记住车钥匙在鞋柜上"→再问准确回答 |
| 9 | 重要事实记忆 | 存储电话/地址等关键事实，自然语言查询 | 说"记住张经理电话138xxxx"→再问报出号码 |
| 10 | 定时提醒 | 到点SSE推送+启动自动补发 | 设提醒→DB有记录→到点通知→重启不丢 |
| 11 | 人物重要度 | 1-5级，≥4自动开启主动收集 | 说"王建国很重要"→FG节点 importance 更新 |

## 六、10分钟快速体检清单

每次改代码后跑这 8 项确认核心链路没崩：

```
□ 1. 发一轮对话 → 刷新页面，对话还在
□ 2. 查数据库 → conversations / memories / 关联ID 都有
□ 3. 设一个 1 分钟后的提醒 → 到点弹出通知
□ 4. 问一个知识库内的问题 → 答案用到了库中内容
□ 5. 说"记住测试物品在桌子上" → 再问能答出来
□ 6. 重启后端 → 历史对话/提醒/记忆全部保留
□ 7. 调用 /api/health/storage → 无红色异常
□ 8. 前端操作上传/撤回/音色切换 → 均正常无报错
```
