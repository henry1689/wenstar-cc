## 文档一：项目总设计规格书 v1.0

### 卷首

### 第一部分：系统灵魂（永久不变）

**第1章 设计哲学与核心范式**
- 1.1 双主体共生的关系范式（vs 传统用户中心范式）
- 1.2 五大核心锚点（DNA身份证、5区异构拓扑、AI自我模型、逆向转译协议、三源融合）
  - 每一条一句话定义 + 一句话"为什么不可妥协"
- 1.3 最小语义单位（MEU）原则
  - 定义："书页"里存的必须是完整体验，不是文字碎片
  - 三条规则：时间间隔>30min、话题切换、情感翻转
  - 为什么不用 token 数 / 消息条数 / 句子数做切割
- 1.4 规则优先 vs 模型黑盒
  - 编码阶段零 LLM（L0纯规则、L3规则NER）
  - 何时可以用 LLM（M5 逆向转译），如何隔离
- 1.5 系统边界声明
  - 系统不做什么（举例：不允许多模态输入、不接入实时语音、不开放第三方API接入）

**第2章 全局架构蓝图**
- 2.1 系统分层图（ASCII 架构图）
  ```
  输入层 → [语义边界检测] → DNA编码层 → 存储层 → 认知组装 → 策略选择 → 语言生成
                 M1              M1        M2-M4     M5(part)    M5(part)    M5(part)
  ```
- 2.2 数据流全景
  - 写入流：原始输入 → L0(L1(L2(L3))) → 5区持久化
  - 提取流（M3+）：L0定位 → L1排序 → L2并行提取 → L3增强
  - 表达流（M5）：DNA语义流 + 知识补丁 → 认知组装 → 策略选择 → 语言转译 → 人文校准
- 2.3 核心技术约束
  - 性能红线：单次编码 ≤30ms（M1），还原 ≤100ms（M3），认知组装 ≤20ms
  - 冷启动策略：出厂默认自我模型（self_model_v1.json）
  - 会话状态生命周期：session 级，resetSession() 时清空

### 第二部分：模块规格（动态更新）

**第3章 M1 DNA编码器（已确权）**
- 3.1 职责定义与边界
  - 输入：用户话语 / 时间戳 / 自我模型
  - 输出：完整DNA对象（内存态，不持久化）
  - M1-M2边界：逻辑生成 vs 物理存储分离
- 3.2 内部架构
  - L0Router（确定性规则路由）：taxonomy分类树 + 关键词匹配 + misc兜底
  - L1Sequencer（branch_id + seq_pos）：`evt_YYYYMMDD_NNN` 格式
  - L2ContentExtractor（leaf_zone + ref）：话题→语义区映射
  - L3EntityAnnotator（NER + phenotype）：规则NER + 情感极性判断 + knowledge_type分类
  - SemanticBoundaryDetector：三条边界规则 + 字符级 Jaccard 相似度兜底
  - DNAEncoder：L0→L1→L2→L3 顺序流水线 + push/flush 流式缓冲
- 3.3 接口契约
  - `push(input: string | PushInput): DNA | null`
  - `flush(): DNA | null`
  - `encodeSingle(utterance, context?): DNA`
  - `encodeBatch(inputs): DNA[]`
- 3.4 关键决策与偏差说明
  - 决策①：L0纯规则（vs 原讨论中的LLM方案）
  - 决策②：编码时 taxonomy 版本号绑定（支持未来记忆重构）
  - 决策③：self_model 构造函数注入（vs 每次传参）
  - 偏差列表：当前实现与白皮书原文的差异及原因

## 文档二：执行任务协议 v2.0

### 1. 协作流程（SOP）

**1.1 标准开发闭环**
```
需求/任务明确 → 复述核心约束 → 设计（含接口定义）→ 评审 → 编码
→ 自检自修 → Hook验证 → 提交 → 总体验收
```

**1.2 强制卡点**
| 卡点 | 触发条件 | 违规处理 |
| :--- | :--- | :--- |
| 无设计不编码 | 开始编码前未输出设计文档 | 拒绝进入编码 |
| 无Hook不提交 | 未编写该模块的 Validation Hook | Hook 补齐前不验收 |
| 无文档不验收 | 模块交付时未更新规格书对应章节 | 不签字通过 |
| 3次自修失败必报告 | 连续3次自修仍不过 | 暂停，报告人工 |

### 2. 架构决策记录（ADR）规范
- 文件路径：`docs/adr/` 目录
- 格式：`ADR-001-l0-pure-rule.md`
- 模板：决策标题 / 状态 / 背景 / 选项 / 决策 / 理由 / 后果

### 3. AI Agent 行为准则
- 3.1 遇到模糊需求时的处理流程（先问不猜）
- 3.2 代码注释规范（必须引用规格书章节号）
- 3.3 禁止行为清单
  - 禁止绕开自检自修闭环
  - 禁止在未确认的情况下假设设计意图
  - 禁止"先写了再说"的编码习惯
- 3.4 需要人工确认的场景（L3梦境算法、M6自我演化逻辑等）

### 4. 资产交付标准
- 4.1 文件路径规范
  ```
  src/m{N}/         — 模块代码
  hooks/            — Validation Hooks
  docs/             — 规格书 + 协议
  docs/adr/         — 架构决策记录
  ```
- 4.2 代码规范
  - 必须有类型定义（禁止 `any`）
  - 公共方法必须有 JSDoc
  - 关键逻辑必须有章节引用注释
- 4.3 测试红线
  - 单元测试覆盖率 ≥ 90%（分支覆盖率）
  - 性能测试全部通过
  - 降级测试覆盖所有已知异常路径

### 5. 变更控制
- 5.1 代码与文档冲突时 → 以规格书为准，修改代码
- 5.2 规格书需要变更时 → 发起 ADR，经审阅后修改规格书 + 更新代码
- 5.3 紧急修复可先改代码，但必须在 24 小时内补齐规格书更新
