## 二、评审意见与优化补充

> **评审评分**: 8.5/10 — 定位精准、落地性强、覆盖全面
> **核心补充方向**: ①基建预留 ②全链路可观测 ③节奏精细化

### 补充1：全链路编码与溯源体系（纳入改造主线）

**必要性**: 当前无统一时序主键，记忆/情感/人格/地标数据无法反向溯源到原始输入，既是排障痛点，也是天权系统最核心的底层依赖。

**落地位置**: 纳入第二阶段末尾，与 `schema_version` 表、字段名统一同步实施——合并为一次数据库迁移。

**核心收益**: 一次改造同时完成"全链路可溯源+天权时序主键打底"两个目标。

### 补充2：架构统一阶段的三步顺序

蓝皮书原本第三阶段6项平铺，调整为「底座→核心→上层」的顺序：

```
第一步：基础设施统一（Phase 3a）
  ├─ 全局事件总线统一：Engine EventBus为全系统唯一标准
  ├─ 全局配置中心雏形：收拢各模块硬编码
  └─ 编码体系落地：DNA根码全链路透传

第二步：核心能力统一（Phase 3b）
  ├─ 双24D统一（Engine为主，M3降级为输入预处理）
  ├─ M6人格→Engine情感基线联动
  └─ 实体关系体系对齐

第三步：上层链路打通（Phase 3c）
  ├─ M5角色路由解耦到编排器
  ├─ 场景状态合并
  └─ GenerationOrchestrator接通M5
```

### 补充3：可观测体系（第二阶段配套项）

| 措施 | 说明 |
|------|------|
| 核心指标埋点 | 各模块耗时/成功率/缓存命中率/数据增速 |
| 统一日志规范 | 所有错误/丢弃/降级操作必须带根码与模块标识 |
| 健康度面板 | `/api/modules`补充核心健康指标，替代纯数量统计 |

### 补充4：P0止血分两批次

| 批次 | 内容 | 回归风险 | 时间 |
|------|------|----------|------|
| **第一批次（先上线）** | B01(M9门控)、B02(双重发射)、A01(M8写入)、A02(M7归纳)、C04(伤疤计数)、D03(黑钻路径) | 全部修复错误，不新增逻辑，回归成本低 | 1天 |
| **第二批次（紧随）** | D01(删BoundaryManager)、D05(接Reranker) | 清理与能力激活，不影响现有逻辑 | 0.5天 |
